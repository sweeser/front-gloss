1. Как задать имя пользователя и адрес электронной почты

Имя пользователя необходимо для привязки коммитов к вашему имени. Это не то же самое, что имя пользователя вашей учётной записи GitHub, которое используется для входа в профиль на GitHub. Задать или изменить имя пользователя можно с помощью команды git config. Новое имя будет автоматически отображаться в последующих коммитах, отправленных на GitHub через командную строку. Если хотите скрыть своё реальное имя, можно использовать произвольный набор символов в качестве имени пользователя Git.

Пример кода:

git config --global user.name "John Smith"

Также командой git config можно изменять адрес электронной почты, привязанный к вашим коммитам Git. Новый адрес электронной почты будет автоматически отображаться во всех дальнейших коммитах, отправленных на GitHub через командную строку.

Пример кода:

git config --global user.email "jSmithexamlgmail.com"

2. Кэширование учётных данных

Для кэширования учётных данных можно использовать команду git config с флагом --global. Это позволит вам избежать необходимости вручного ввода имени пользователя и пароля при создании нового коммита.

Пример кода:

git config --global credential.helper cache

3. Инициализация репозитория

С помощью команды git init можно создать пустой репозиторий Git или вновь инициализировать уже существующий. При инициализации будет создана скрытая папка, в которой будут храниться все объекты и ссылки, используемые и создаваемые Git для ведения истории работы над проектом.

Пример кода:

git init

4. Добавление отдельных файлов или всех файлов в область подготовленных файлов

Чтобы добавить отдельный файл в область подготовленных файлов, используйте команду git add, указав имя файла. Например, замените somefile.js на имя вашего файла:

Пример кода:

git add somefile.js

Чтобы добавить все файлы и папки в область подготовленных файлов, используйте символ подстановки точки вместо имени файла:

Пример кода:

git add .

5. Проверка статуса репозитория

Чтобы просмотреть статус репозитория, используйте команду git status. Эта команда покажет информацию о подготовленных, неподготовленных и неотслеживаемых файлах.

Пример кода:

git status

6. Внесение изменений однострочным сообщением или через редактор

При создании коммита в репозитории можно добавить однострочное сообщение с помощью команды git commit и флага -m. Сообщение вводится в кавычках непосредственно после флага.

Пример кода:

git commit -m "Your text for commit"

Также можно открыть текстовый редактор в терминале для написания полного сообщения коммита, которое может состоять из нескольких строк и подробно описывать внесённые изменения.

Пример кода:

git commit

7. Просмотр истории коммитов с изменениями

Чтобы просмотреть изменения, внесённые в репозиторий, используйте команду git log. Она отображает список последних коммитов в порядке их выполнения. Для подробного изучения изменений в каждом файле добавьте флаг -p.

Пример кода:

git log -p

8. Просмотр заданного коммита

Чтобы просмотреть полный список изменений, внесённых конкретным коммитом, используйте команду git show с указанием идентификатора или хеша коммита. Хеш уникален для каждого коммита в вашем репозитории.

Пример кода:

git show 1af17e73721dbe0c40011b82ed4bb1a7dbe3ce29

Вы также можете использовать сокращённый хеш.

Пример кода:

git show 1af17e

9. Просмотр изменений до коммита

Для просмотра списка изменений, внесённых в репозиторий, используйте команду git diff. По умолчанию отображаются только изменения, не подготовленные для коммита.

Пример кода:

git diff

Чтобы просмотреть подготовленные изменения, добавьте флаг --staged.

Пример кода:

git diff --staged

Также можно указать имя файла, чтобы просмотреть изменения, внесённые только в этот файл.

Пример кода:

git diff somefile.js

10. Удаление отслеживаемых файлов из текущего рабочего дерева

Для удаления файлов из текущего рабочего дерева и индекса используйте команду git rm. Например, чтобы удалить конкретный файл.

Пример кода:

git rm dirname/somefile.js

Вы также можете использовать маски файлов для удаления всех файлов, соответствующих определённому критерию.

Пример кода:

git rm dirname/*.html

11. Переименование файлов

Для переименования файла или папки используйте команду git mv, указав источник (source) и назначение (destination). Источник — это реально существующий файл или папка, а назначение — существующая папка.

Пример кода:

git mv dir1/somefile.js dir2

При выполнении этой команды файл или папка, указанные как источник, будут перемещены в папку назначения. Индекс будет обновлён соответственно, но изменения необходимо зафиксировать.

12. Отмена подготовленных и неподготовленных изменений

Чтобы восстановить файлы рабочего дерева, не подготовленные к коммиту, используйте команду git checkout с указанием пути к файлу. Если путь не указан, команда git checkout изменит указатель HEAD, чтобы задать указанную ветку как текущую.

Пример кода:

git checkout somefile.js

Для восстановления подготовленного файла рабочего дерева используйте команду git reset. Укажите путь к файлу, чтобы убрать его из области подготовленных файлов. Это не откатит изменения, а просто переведёт файл в категорию неподготовленных к коммиту.

Пример кода:

git reset HEAD somefile.js

Чтобы выполнить это действие для всех подготовленных файлов, путь указывать не нужно.

Пример кода:

git reset HEAD

13. Изменение последнего коммита

Для внесения изменений в последний коммит используйте команду git commit с флагом --amend. Например, если вы записали изменения в несколько файлов и обнаружили ошибку в сообщении коммита, вы можете отредактировать сообщение предыдущего коммита, не изменяя его содержимое.

Пример кода:

git commit --amend -m "Updated message for the previous commit"

Также можно внести изменения в файлы, включённые в предыдущий коммит. Например, если вы изменили несколько файлов и хотите зафиксировать их как единый снимок, но забыли добавить одну из папок, можно подготовить забытые файлы и создать коммит с флагами --amend и --no-edit.

Пример кода:

git add dir1
git commit

# Если забыли добавить dir2, выполните следующие команды для внесения изменений:

git add dir2
git commit --amend --no-edit

Флаг --no-edit позволяет внести изменения в коммит без изменения сообщения. Итоговый коммит заменит предыдущий, создавая впечатление, что все изменения были зафиксированы как единый снимок.

Важно: Не изменяйте публичные коммиты.

Использование --amend отлично подходит для исправления локальных коммитов, которые затем можно передать в общий репозиторий. Однако, не следует изменять коммиты, уже доступные другим пользователям. Изменённые коммиты считаются новыми, и предыдущий коммит станет недоступен в текущей ветке, что может привести к таким же последствиям, как отмена изменений публичного снимка.

14. Откат последнего коммита

Чтобы откатить последний коммит, используйте команду git revert. Эта команда создаёт новый коммит, содержащий изменения, отменяющие предыдущий коммит, и добавляет его к истории текущей ветки.

Пример кода:

git revert HEAD
Разница между revert и reset

Команда git revert отменяет изменения, записанные только одним коммитом. Она не откатывает проект к более раннему состоянию, удаляя все последующие коммиты, как это делает команда git reset.


Преимущества revert по сравнению с reset:
  1. Сохранение истории проекта: git revert не изменяет историю проекта, что делает операцию безопасной для коммитов.
  2. Отмена конкретного коммита: git revert позволяет отменить конкретный коммит, созданный в любой момент истории. В отличие от этого, git reset всегда берёт за точку отсчёта текущий коммит.

Например, чтобы отменить старый коммит с помощью git reset, нужно удалить все коммиты, сделанные после целевого, а затем выполнить их повторно. Поэтому команда git revert является более удобным и безопасным способом отмены изменений.

15. Откат заданного коммита

Чтобы откатить изменения, внесённые конкретным коммитом, используйте команду git revert с указанием идентификатора коммита. Это создаст новый коммит, который отменит изменения, сделанные указанным коммитом, и добавит его к истории текущей ветки:

Пример кода:

git revert 1af17e

Эта команда отменяет изменения, внесённые указанным коммитом, но не удаляет сам коммит из истории. В результате создаётся новый коммит, который является обратным преобразованием указанного коммита.

16. Создание новой ветки и переход в неё

Чтобы создать новую ветку, используйте команду git branch с указанием имени новой ветки:

Пример кода:

git branch new_branch_name

Однако, после создания новой ветки Git не переключится на неё автоматически. Чтобы создать новую ветку и сразу же перейти в неё, используйте команду git checkout с флагом -b.

Пример кода:

git checkout -b new_branch_name

17. Просмотр списка веток

Чтобы просмотреть полный список веток, используйте команду git branch. Эта команда отобразит все ветки, отметит текущую звёздочкой (*) и выделит её цветом,

Пример кода:

git branch

Чтобы вывести список всех веток, включая удалённые, используйте флаг -a.

Пример кода:

git branch -a

18. Удаление ветки

Чтобы удалить ветку, используйте команду git branch с флагом -d и указанием имени ветки. Если работа над веткой завершена и она была объединена с основной, можно удалить её без потери истории. Если попытаться удалить ветку до её слияния, появится сообщение об ошибке, предупреждающее о возможной потере данных.

Пример кода:

git branch -d existing_branch_name

Для принудительного удаления ветки, независимо от её текущего статуса, используйте флаг -D (заглавная буква). В этом случае ветка будет удалена без предупреждений.

Пример кода:

git branch -D existing_branch_name

Эти команды удаляют только локальную копию ветки. Чтобы удалить ветку в удалённом репозитории, используйте следующую команду.

Пример кода:

git push origin --delete existing_branch_name

19. Слияние двух веток

Чтобы объединить две ветки, используйте команду git merge с указанием имени ветки, которую нужно объединить с текущей:

Пример кода:

git merge existing_branch_name

Если необходимо выполнить коммит слияния, используйте команду git merge с флагом --no-ff.

Пример кода:

git merge --no-ff existing_branch_name

Эта команда объединит указанную ветку с текущей и произведёт коммит слияния. Это полезно для фиксации всех слияний в вашем репозитории, сохраняя историю изменений.

20. Отображение журнала фиксации в виде графика для текущей или всех веток

Чтобы просмотреть историю коммитов в виде графика для текущей ветки, используйте команду git log с флагами --graph, --oneline, и --decorate. Опция --graph выведет график в формате ASCII, показывающий структуру ветвления истории коммитов. В сочетании с флагами --oneline и --decorate эта команда упрощает понимание того, к какой ветке относится каждый коммит.

Пример кода:

git log --graph --oneline --decorate

Для просмотра истории коммитов по всем веткам добавьте флаг --all.

Пример кода:

git log --all --graph --oneline --decorate

21. Прекращение слияния при конфликте

Чтобы прервать слияние в случае конфликта, используйте команду git merge с флагом --abort. Это остановит процесс слияния и вернёт состояние репозитория к тому, с которого этот процесс был начат.

Пример кода:

git merge --abort

Кроме того, при конфликте слияния можно использовать команду git reset, чтобы восстановить конфликтующие файлы до стабильного состояния:

Пример кода:

git reset

Эти команды помогут справиться с конфликтами и вернуть репозиторий в стабильное состояние, позволяя начать процесс слияния заново или принять другие меры для разрешения конфликта.

22. Добавление удалённого репозитория

Чтобы добавить удалённый репозиторий, используйте команду git remote add, указав короткое имя (shortname) и URL репозитория.

Пример кода:

git remote add awesomeapp https://github.com/someurl..

Эта команда связывает удалённый репозиторий с вашим локальным репозиторием, позволяя вам работать с ним под указанным коротким именем.

23. Просмотр удалённых URL-адресов

Чтобы просмотреть удалённые URL-адреса, используйте команду git remote с флагом -v. Эта команда отображает удалённые подключения к другим репозиториям.

Пример кода:

git remote -v

Эта команда предоставляет доступ к интерфейсу управления удалёнными записями, которые хранятся в файле .git/config вашего репозитория.

24. Получение дополнительных сведений об удалённом репозитории

Чтобы получить подробные сведения об удалённом репозитории, используйте команду git remote show с указанием имени репозитория, например, origin.

Пример кода:

git remote show origin

Эта команда отображает список веток, связанных с удалённым репозиторием, а также информацию о рабочих станциях, подключённых для получения и отправки файлов.

25. Отправка изменений в удалённый репозиторий

Чтобы отправить изменения в удалённый репозиторий, используйте команду git push с указанием имени репозитория и ветки.

Пример кода:

git push origin main

Эта команда передаёт локальные изменения в центральный репозиторий, позволяя другим участникам проекта ознакомиться с ними.

26. Получение изменений из удалённого репозитория

Для загрузки изменений из удалённого репозитория используйте команду git pull. Эта команда скачивает копию текущей ветки из указанного удалённого репозитория и объединяет её с локальной копией.

Пример кода:

git pull

Чтобы просмотреть подробные сведения о загруженных файлах, используйте флаг --verbose.

Пример кода:

git pull --verbose

Эта команда предоставляет более детальную информацию о процессе загрузки и объединения изменений.

27. Слияние удалённого репозитория с локальным

Чтобы слить изменения из удалённого репозитория с локальным, используйте команду git merge с указанием имени удалённого репозитория.

Пример кода:

git merge origin

Эта команда объединяет изменения из удалённого репозитория с текущей локальной веткой.

28. Отправка новой ветки в удалённый репозиторий

Чтобы отправить новую ветку в удалённый репозиторий, используйте команду git push с флагом -u, указав имя репозитория и имя ветки.

Пример кода:

git push -u origin new_branch

Флаг -u устанавливает upstream (отслеживание) для указанной ветки, что упрощает последующую работу с этой веткой в удалённом репозитории.

29. Удаление удалённой ветки

Чтобы удалить ветку в удалённом репозитории, используйте команду git push с флагом --delete, указав имя удалённого репозитория и имя ветки.

Пример кода:

git push --delete origin existing_branch

Эта команда удалит указанную ветку из удалённого репозитория.

30. Использование перебазирования

Чтобы использовать перебазирование, примените команду git rebase с указанием имени ветки. Перебазирование — это процесс объединения или перемещения последовательности коммитов на новый родительский снимок.

Пример кода:

git rebase

Эта команда изменяет основу ветки с одного коммита на другой, как если бы вы начали ветку с другого коммита. В Git это достигается путём создания новых коммитов и их применения к указанному базовому коммиту. Важно понимать, что хотя ветка будет выглядеть такой же, она будет состоять из совершенно новых коммитов.
